01 · 审阅结论

OK可以继续做，不需要推倒重来

当前工具链已经把 NSF 播放问题降维成“按帧回放 APU 寄存器写入”。这正适合课程项目：Logisim 里不用实现 6502、NSF loader、真实混音器，只需要做一个能读指令 ROM 的简化 CPU 和一个能把寄存器翻译成 Buzzer 输入的简化 APU。

DO NOT不要按原始 v6 文案直接开工

原始计划书 `plan/project_brief_v6.html` 已经被后续实现部分推翻：实际 hook 目标是 GME 不是 NSFPlay；PC 必须 16 位；Buzzer 频率吃真实 Hz，不跟 Logisim 时钟缩放；节奏已经由 `encode.py --frame-cycles` 固定化。新的实施以本文件为准，旧计划只当背景材料。

判断项结论理由
技术路线成立离线脚本输出 16-bit ROM；Logisim ROM/Buzzer/Register/Mux 都能覆盖所需组件。
最大正确性风险Buzzer 性能与声音伪真度Logisim Buzzer 每次输入变化会重建音频缓冲并开新 Clip，本地源码还显示 sound thread 是忙循环。
最大工期风险硬件电路尚未开始`coolproject.circ` 只有空的 `main` 电路，脚本完成不等于可演示。
推荐目标先保 1 首、4 声道、五段 CPU、ALU 变速播放 + 可视化 demo真实音乐播放时 ALU 参与 WAIT 周期加减；R0-R3 demo 只作为验收时的直观证明。

02 · 文档依据和本地证据

下面是我实际用于校正计划的依据。Logisim 行为优先看官方文档和本仓库 `ref/logisim-evolution` 源码；NES APU/NSF 用 NESdev 作为权威社区规范；GME 用官方仓库和本地补丁记录。

主题依据对本项目的含义
Logisim BuzzerLogisim-evolution Buzzer 文档；本地 `Buzzer.java`Frequency 输入是 14-bit，单位为 Hz/dHz；支持 square/triangular/white noise；有效频率 20-20000Hz，但 14-bit 总线在 Hz 模式最多表示 16383。
Logisim ClockClock 文档Clock 依赖 Simulate 菜单的 Auto-tick 和 Tick Frequency；真实节奏要靠全局 tick 频率调到约 1920Hz。
ROM / memory image本地 `ref/logisim-evolution/.../mem-v2raw.html` 和 `rom.html`；原 Logisim ROM 文档`v2.0 raw` 第一行后是从地址 0 开始的十六进制值；ROM 可存 2^AddressBitWidth 个值，单值最多 32-bit，本项目用 16-bit 指令字。
NES APU 寄存器NESdev APU registers基础 APU 地址是 `$4000-$4013,$4015,$4017`；`$4015` 的低 4 位控制 noise/triangle/pulse2/pulse1 使能。
Pulse/Triangle/Noise 公式APU Pulse、APU Triangle、APU NoisePulse 频率公式是 `fCPU/(16*(t+1))`；Triangle 是 `fCPU/(32*(t+1))`；Noise 使用 `$400E` 低 4 位查 16 项周期表。
NSF 格式NESdev NSFNSF 是 NES 音乐程序加头；播放器负责装载并周期性调用 INIT/PLAY。本项目离线跑一遍再 dump 寄存器，方向正确。
GMElibgme/game-music-emu；`script/gme_dump/PATCH.md`官方仓库包含 `Nsf_Emu.cpp` 和 `Nes_Apu.cpp/.h`；本项目在 `Nes_Apu::write_register` 和 `Nsf_Emu` 帧调用处加 hook，是比 NSFPlay 更稳的选择。
Java ClipOracle Java Clip APIClip 打开时会加载音频数据，可循环播放；频繁 open/close 有资源限制风险，和 Buzzer 源码风险一致。

05 · 最终架构锁定

.nsf
  -> GME dumper: 每次 PLAY 帧 + 每次 APU write
  -> log: "frame addr value"
  -> encode.py: 16-bit 指令 ROM, 固定 frame_cycles
  -> Logisim CPU: WAIT / WRITE / END + ALU tempo control
  -> Logisim APU: 2 pulse + triangle + noise -> 4 个 Buzzer
  -> 控制面板: play/pause/stop/next/prev/timer

指令集以“硬件最简”为优先

Opcode用途实现优先级
`0x0 WAIT imm12`等待 imm 个 CPU tick必须，注意 busy 状态机。
`0x1 WRITE reg,val`写 APU 寄存器必须，真实音乐只靠它和 WAIT/END 就能跑。
`0x2 LWAIT + next word`长等待做译码入口即可；当前 ROM 基本不需要。
`0x4..0x7 LOADI/ADD/SUB/WRITEREG`显式 ALU 指令和寄存器演示用于把 ALU 算术路径可视化；真实播放则通过 WAIT 变速路径持续使用 ALU。
`0xF END`曲末 PC 清 0 或自动下一首必须。

APU 寄存器映射

继续使用 `encode.py` 的 4-bit `reg_id` 映射。`$4015` 放不进 `$4000-$400F` 的 16 槽，所以借真实未用的 `$4009` 槽，也就是 `reg 9`。

reg_id含义原 NES 地址
0-3Pulse 1`$4000-$4003`
4-7Pulse 2`$4004-$4007`
8,10,11Triangle`$4008,$400A,$400B`
9声道使能`$4015` 重映射，低 4 位 = N/T/P2/P1
12,14,15Noise`$400C,$400E,$400F`
13不用`$400D` 真实未用，脚本丢弃

ALU 与播放器的结合点：实时变速

不要把 ALU 做成孤立小 demo。主线功能是：CPU 遇到 `WAIT imm` 时，不直接把 `imm` 装进 `wait_counter`，而是先让 ALU 计算有效等待时间。

normal : effective_wait = imm
slower : effective_wait = imm + tempo_delta
faster : effective_wait = max(1, imm - tempo_delta)

`tempo_delta` 由控制面板的快/慢按钮或 2-bit 档位选择产生，例如 0、2、4、8。这样真实 NSF 播放时，每一帧的 WAIT 都会经过 ALU；老师问 ALU 有什么用，可以直接演示“慢一点/快一点”按钮改变播放速度。R0-R3 的 demo ROM 保留，但它只是验证 ALU 加减的可视化辅助。

06 · 施工顺序：照这个做

  1. 先做 Buzzer 压测小电路。不要等 CPU/APU 搭完才发现声音线程卡。做一个 `BuzzerStress` 子电路：计数器每 1、2、4、8、16、32 tick 改一次频率，分别测单 Buzzer 和 4 Buzzer。
  2. 做单声道 Pulse APU。手动用常量和按钮写 `$4000,$4002,$4003,$4015`，能让一个 Buzzer 发 A4 或音阶。
  3. 扩成 4 声道 APU。先不追求真实包络，只做 timer、volume、duty、enable，确保 4 个 Buzzer 能同时响。
  4. 做最小 CPU。先实现 `WAIT/WRITE/END`，加载 `track00.txt` 或更短测试 ROM，能自动写 APU 出声。
  5. 把 ALU 接入 WAIT。在 `WAIT` 装载计数器前加 `effective_wait = imm ± tempo_delta`，先用常量档位测试快/慢播放。
  6. 补显式 ALU demo。加入 R0-R3、`LOADI/ADD/SUB/WRITEREG`，用 demo ROM 显示 `100+50=150`、`100-50=50`，证明同一套加减硬件可控。
  7. 补控制面板。做 pause、stop、next、prev，以及 normal/fast/slow 变速档位。多曲 MUX 只有在前面都稳后再搭。
  8. 补计时和展示。MM:SS 和钢琴键可视化是加分项，最后做。

07 · CPU 实现细节

关键WAIT 必须有 busy 状态机

如果 PC 在 WAIT 上停住，而组合逻辑每拍看到 WAIT 都重新装载计数器，就会永远 WAIT。正确做法是加 `wait_busy`：第一次遇到 WAIT 时装载并置 busy；busy 期间只递减；递减到 0 的那一拍清 busy 并让 PC+1。

IF  : PC -> ROM -> instr[15:0]
ID  : opcode = instr[15:12], 拆 imm/reg/val/Rd/Ra/Rb
EX  : WAIT 调速 ALU / 显式 ALU 加减 / reg_id 译码
MEM : WRITE 或 WRITEREG -> APU write bus
WB  : PC 更新、寄存器写回

推荐寄存器和控制信号

名字位宽说明
`PC`16字地址，接当前曲目 ROM 地址。
`instr`16ROM 输出，可直接进 splitter。
`wait_counter`16装载 `effective_wait`，支持 WAIT 和 LWAIT。
`wait_busy`1避免 WAIT 重装载。
`tempo_delta`8 或 16快/慢播放的步长，来自按钮或 DIP 档位。
`tempo_mode`200 正常，01 加长 WAIT 变慢，10 缩短 WAIT 变快。
`R0..R3`8显式 ALU demo 和 WRITEREG 数据源。
`apu_we`1仅 WRITE/WRITEREG 且非 wait stall 时为 1。
`apu_reg`4来自 instr[11:8]。
`apu_data`8WRITE 用 instr[7:0]，WRITEREG 用 R[Rs]。

最小测试 ROM

v2.0 raw
4100 4164 4232 5200 7900 f000

示例含义按你当前编码微调：LOADI R0,100；LOADI R1,50；ADD R2,R0,R1；WRITEREG 某 reg,R2；END。若字段定义和现有表不一致，以本文件第 05 节 ISA 为准重新编码。

WAIT 变速硬件接法

  1. 把 `WAIT imm12` 零扩展到 16-bit，作为 ALU 输入 A。
  2. 把 `tempo_delta` 零扩展到 16-bit，作为 ALU 输入 B。
  3. `tempo_mode=00` 时旁路 ALU，直接装载 `imm`；`01` 时选择 `A+B`；`10` 时选择 `A-B`。
  4. 对快速档加保护：如果 `A <= B`，装载 1，避免 WAIT=0 导致播放过快或忙等逻辑边界不清。
  5. 把最终的 `effective_wait` 显示到一个 Hex Display，验收时按快/慢按钮能看到它变化。

08 · APU 实现细节

先做“可听见”，再做“更像 NES”

通道寄存器频率Volume / Duty / Enable
Pulse 1`reg0..3``timer = reg2 | (reg3[2:0] << 8)` 查 `freq_pulse``vol = reg0[3:0]` 扩到 7-bit；`duty = reg0[7:6]` 映射 32/64/128/192；enable = `reg9[0] & timer>=8`。
Pulse 2`reg4..7``timer = reg6 | (reg7[2:0] << 8)` 查 `freq_pulse`同 Pulse 1；enable = `reg9[1] & timer>=8`。
Triangle`reg8,10,11``timer = reg10 | (reg11[2:0] << 8)` 查 `freq_triangle`Buzzer 设 Triangle，Volume 常量。enable = `reg9[2] & timer>1`，必要时再加 `reg8[6:0] != 0`。
Noise`reg12,14,15``reg14[3:0]` 查 `freq_noise`Buzzer 设 White noise；Volume = `reg12[3:0]` 扩到 7-bit；enable = `reg9[3]`。

风险Buzzer 源码显示它不是轻量振荡器

本地 `Buzzer.java` 证据：`propagate()` 每次都会 `updateRequired=true`；sound thread 在更新时会创建 `double[sampleRate]` 和 `byte[4*sampleRate]`，再 `AudioSystem.getClip()`、`open()`、`loop()`；源码还在 `startThread()` 注释里警告把 4kHz clock 接 enable 会崩。更麻烦的是 `while (isOn.get())` 里没有 sleep，静态长音也可能占 CPU。因此必须先压测。

压测标准

  • 1 个 Buzzer：30Hz、60Hz、120Hz 频率更新都能稳定响。
  • 4 个 Buzzer：60Hz 更新不明显卡顿；CPU 占用可接受。
  • Logisim 全局 tick 调到 1920Hz 左右时，按钮和计数器仍然响应。
  • 如果卡，先把 APU 输出前加采样寄存器，只每 2-4 帧更新一次 Buzzer；其次才考虑删声道。

11 · 验收时怎么讲

  1. 先讲降维：NSF 本质是 NES 音乐程序，离线播放器跑它并记录 APU 寄存器写入，Logisim 只回放寄存器流。
  2. 再讲 CPU：16-bit 定长 ISA，五段 IF/ID/EX/MEM/WB，WAIT 是多周期状态，WRITE 是 MMIO。
  3. 展示 ALU 主功能：播放真实 `track00.txt`，切 normal/slow/fast，说明 WAIT 装载前会经 ALU 计算 `imm ± tempo_delta`，所以真实音乐播放时 ALU 持续参与节奏控制。
  4. 展示 ALU 旁证：现场跑 demo ROM，让 R0-R3 的 Hex Digit 变化，说明 ADD/SUB 是同一类可控算术路径。
  5. 展示 APU：Pulse/Triangle/Noise 分别来自 NESdev 公式和查表 ROM，Buzzer 负责最终发声。
  6. 主动说明取舍：DMC、真实包络、真实长度计数器、真实混音暂不做；这是课程范围内的工程简化，不是没想清楚。
