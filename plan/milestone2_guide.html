里程碑 2：让寄存器来驱动声音 —— 真正的单 Pulse 通道 里程碑 1 你用手直接喂 timer 让 Buzzer 出声。本里程碑把“手喂”换成“写寄存器”：搭一个寄存器前端（译码器 + 寄存器组）+ 一个可复用的 Channel_Pulse 子电路，从 $4000/$4002/$4003/$4015 解码出音高 / 音量 / 占空比 / 使能。做完你会得到一个用 NES 寄存器协议驱动的、可复用的单通道，并顺手把它扩成 4 路、封进 APU 子电路——这正是接 CPU 前的最后一块拼图。

里程碑 1 的结论（已实测，作为本里程碑的地基）
  • 频率表 + 蜂鸣器（Buzzer）链路音高正确：freq_pulse[253] = 440Hz = A4，手喂 timer 能弹音阶。
  • 蜂鸣器（Buzzer）扛得住频率变化（用分线器（Splitter）取计数器（Counter）高位分频后，扫频流畅）。
  • 核心原理（本里程碑要继续遵守）：蜂鸣器（Buzzer）每次输入变化都会重建音频缓冲，所以绝不能让高速时钟（Clock）每一拍都改它的输入。换音率应当压在音乐帧率 ~60Hz。本里程碑会专门加一道“按帧采样”门控来落实这一点（见 §4.5）。

0这一里程碑要搭什么

从“手喂 timer”到“写寄存器”

真实 NES 的 CPU 不会直接给声卡一个频率，它往寄存器里写字节：把音高拆成 $4002(低 8 位)+$4003(高 3 位)，把音量/占空比塞进 $4000，用 $4015 开关声道。APU 再把这些寄存器值“翻译”成频率/音量喂给发声单元。本里程碑就是在 Logisim 里复刻这套“寄存器 → 翻译 → 蜂鸣器（Buzzer）”的链路。

两个新子电路（都建在 apu.circ 里）
apu.circ
├── BuzzerStress      （里程碑 1，留作回归测试，不动）
├── PulseOneShot      （里程碑 1，留作对照，不动）
├── RegFile      ★新建：寄存器前端 = 译码器(Decoder) + 4 个寄存器(Register)
│                       对外像 CPU 的写口：reg_id(4) / value(8) / WR(1) 进，寄存器值出

   （§5 再把 Channel_Pulse 复制/改造成 Pulse2 / Triangle / Noise，最后封一个 APU 顶层）

分两层的好处：RegFile 是所有通道共享的“写总线”（CPU 以后只跟它打交道）；Channel_Pulse 是纯“翻译+发声”，不关心写入时序。这样 4 个通道能共用一套寄存器，答辩时也好讲：“写口就这 3 根线”。

2寄存器前端 RegFile：译码器 + 寄存器组

先建“写口”。它接收 reg_id / value / WR，把 value 锁进 reg_id 选中的那个寄存器。本里程碑只需 Pulse1 用到的 4 个寄存器：reg0=$4000、reg2=$4002、reg3=$4003、reg9=$4015（声道使能；按计划 $4015 重映射到 reg_id 9）。

R1

新建子电路 RegFile

确认在编辑 apu.circ。项目 → 添加电路…（Project → Add Circuit…），名字 RegFile，双击进入。

R2

放 3 个输入 引脚（Pin）当“写口”

线路 → 引脚（Wiring → Pin），放三个，分别选中改属性：
• 类型（Type）=Input，数据位宽（Data Bits）=4，标签（Label）=reg_id
• 类型（Type）=Input，数据位宽（Data Bits）=8，标签（Label）=value
• 类型（Type）=Input，数据位宽（Data Bits）=1，标签（Label）=WR（这根当“写脉冲/时钟”用）

R3

放 译码器（Decoder），设 4 选择位 → 16 路

复用器 → 译码器（Plexers → Decoder）。选中它：选择位数（Select Bits）=4（输出变成 16 路）。把 reg_id 接到它的选择脚。
若它带“使能”脚（属性 启用使能（Include Enable?）=是）：要么把该属性改成 否，要么给使能脚接一个 值=1 的 常量（Constant），否则整个译码器被禁用、所有输出为 0。

R4

放 4 个 寄存器（Register），各 8 位

存储器 → 寄存器（Memory → Register） 放 4 个，每个 数据位宽（Data Bits）=8，触发方式（Trigger） 保持上升沿。建议各贴 标签（Label）：reg0、reg2、reg3、reg9。

R5

接线：value→D，WR→时钟，译码器输出→写使能

对这 4 个寄存器：
• value(8-bit) 同时接到每个寄存器的 D（数据入）。一根总线分叉到 4 处即可。
• WR(1-bit) 同时接到每个寄存器的时钟脚(▷)。
• 译码器（Decoder）的第 0 路输出 → reg0 的写使能；第 2 路 → reg2；第 3 路 → reg3；第 9 路 → reg9。（其余 12 路输出悬空不接。）
• 每个寄存器（Register）的清除脚接一个 1 位、值为 0 的 常量（Constant）。不要让清除脚飘着；源码里 CLR==1 会异步清零，接 0 最稳。

为什么这样接就对

WR 上升沿到来时，4 个寄存器都收到时钟边沿，但只有写使能=1 的那个会真正把 value 锁进去。译码器保证“同一时刻最多一路使能=1”（reg_id 选中谁，谁的使能才是 1）。于是 reg_id=2, value=0xFD, 点一下 WR ⇒ 只有 reg2 变成 0xFD。这正是 CPU WRITE reg,val 的硬件含义。

R6

放 4 个输出 引脚（Pin），把寄存器值引出去

再放 4 个 引脚（Pin），类型（Type）=Output、数据位宽（Data Bits）=8，标签（Label） 依次取 reg0_out、reg2_out、reg3_out、reg9_out（务必带 _out 后缀，原因见下方红框），分别接到对应 寄存器（Register）的 Q。这样 RegFile 作为子电路被实例化时，外面就能拿到这 4 个寄存器的当前值。保存文件。

为什么输出脚叫 reg0_out 而不是 reg0

同一个子电路里，标签（Label） 必须唯一——和部件类型无关。R4 的 4 个 寄存器（Register）已经占用了 reg0/reg2/reg3/reg9 这几个名字；若输出 引脚（Pin）再用同名，即便一个是 寄存器（Register）、一个是 引脚（Pin），Logisim 也会判为重复并弹框拒绝：「已存在使用该标签的组件！请在电路中使用唯一的标签。」（There exists already a component with this label! Please use unique labels in a circuit.）。所以输出统一加 _out 后缀避让；本项目后面所有子电路的「寄存器值输出脚」都照此命名（regN_out）。
来源：ref/logisim-evolution/.../circuit/Circuit.java 的 isExistingLabel()——它遍历电路内全部组件、只比对 标签（Label） 字符串（忽略大小写，且 隧道（Tunnel）除外），完全不看部件种类。

线太多？用 隧道（Tunnel）

value 要分叉到 4 个 D、WR 要分叉到 4 个时钟——画起来容易乱。可用 线路 → 隧道（Wiring → Tunnel）：给一根线起个名字（如 VAL），别处放同名 隧道（Tunnel）就等于连上了，不用真的拉线。§5 扩到 4 通道时这招能救命。

3.1 拼出 11 位 timer（本里程碑的关键动作）

C2

第一个 Splitter（分线器）：从 reg3 里取出低 3 位

放一个 Splitter（分线器）。属性：输入位宽（Bit Width In）=8、分支数（Fan Out）=2。
选中后属性表会列出 Bit 0 … Bit 7 各归到哪个分支：把 Bit 0、1、2 设为分支 0（这就是我们要的低 3 位），Bit 3–7 设为分支 1（丢弃不接）。
把 reg3 接到 Splitter 的合并端（那一根 8 位侧）；分支 0 引出来就是 reg3[2:0]（3 位）。

C3

第二个 Splitter（分线器）：把 reg2(8) + reg3低3(3) 拼成 timer(11)

再放一个 Splitter（分线器），这次当“合并器”用。属性：输入位宽（Bit Width In）=11、分支数（Fan Out）=2。位分配：Bit 0–7 → 分支 0（低 8 位），Bit 8–10 → 分支 1（高 3 位）。
接线：reg2(8 位) → 分支 0；上一步的 reg3[2:0](3 位) → 分支 1。合并端引出的 11 位总线就是 timer = (reg3[2:0]<<8) | reg2。

位宽必须严丝合缝

分支 0 必须正好 8 位、分支 1 正好 3 位，合起来 11 位。线是黑色（多位总线）才对，灰色 = 位宽没对上（里程碑 1 讲过）。Splitter 的方向/朝向若让接线别扭，调 朝向（Facing） 即可。

3.3 音量 / 占空比 / 使能（从 reg0、enable 解出来）

C5

第三个 Splitter（分线器）：把 reg0 拆成音量位和占空比位

放一个 Splitter（分线器）：输入位宽（Bit Width In）=8、分支数（Fan Out）=3。位分配：Bit 0–3 → 分支 0（音量，4 位）、Bit 4–5 → 分支 1（丢弃）、Bit 6–7 → 分支 2（占空比，2 位）。reg0 接合并端。

C6

音量：reg0[3:0] → 蜂鸣器（Buzzer）的 VOL

放 蜂鸣器（Buzzer）（输入/输出（扩展）→ 蜂鸣器（Input/Output Extra → Buzzer）），波形（Waveform）=Square（方波）。把它的 音量位宽（Volume Width） 改成 4，然后把 reg0 分线器的分支 0（4 位音量）接到 VOL 脚。这样寄存器里 0–15 的音量直接生效（NES Pulse 音量本就是 4 位）。

C7

占空比：reg0[7:6] → 4 项查找 → PW 可后补

NES 占空比 2 位选 4 档 {12.5%,25%,50%,75%}，对应蜂鸣器（Buzzer）8 位 PW 的 {32,64,128,192}={0x20,0x40,0x80,0xc0}。做一个 4 项查找 ROM：
放一个 ROM，地址位宽=2、数据位宽=8，右键 编辑内容…（Edit Contents…） 填 4 个值 20 40 80 c0。把 reg0 分线器的分支 2（2 位占空比）接 ROM 的 A，ROM 的 D(8) 接蜂鸣器（Buzzer）的 PW 脚。
想先简单点：这一步可暂时跳过——PW 直接接一个 值=0x80(50%) 的 8 位 常量（Constant），等整条链路出声后再回来接占空比。

C8

使能：enable → 蜂鸣器（Buzzer）的 ENABLE

把输入 引脚（Pin）enable(1 位) 接到蜂鸣器（Buzzer）的 ENABLE 脚。enable=1 出声、=0 静音——这就是 $4015 声道开关的硬件落点。

C9

FREQ 接好，存盘

把 freq_pulse ROM 的 D(14 位) 接到蜂鸣器（Buzzer）的 FREQ 脚（两端都 14 位，黑线）。至此 Channel_Pulse 内部完工：4 个输入 引脚（Pin）进，一个蜂鸣器（Buzzer）出。保存文件。

  • 音高对：写 reg2/reg3 改 timer，验证“数值越小、音越高”（参考里程碑 1 的表：253→A4、190→D5、126→A5、63→A6；这里 reg2=低 8 位，timer<256 时 reg3 都写 0）。
  • 音量可控：把 reg0 改成 88(音量 8)、81(音量 1)，音量随之变小；80(音量 0) 几乎无声。
  • 占空比可控（若接了 C7）：reg0 写 0F/4F/8F/CF ⇒ 12.5/25/50/75%，音色（明亮↔圆润）有变化。
  • 使能可控：reg9 写 00 立即静音、写 01 复响。

四条都过 = 寄存器协议 → 声音这条链路打通。这是接 CPU 的前提（CPU 无非就是自动地、按曲谱去做你刚才手动做的那串“写 reg_id/value + WR”）。

4.5★ 按帧采样门控：自动扫表/接 CPU 前的保护

为什么现在就要加这道门

§4 你是用手慢慢写寄存器，所以蜂鸣器（Buzzer）输入变化很慢、毫无压力。但一旦用计数器或 CPU 自动写寄存器，上游可能在很短时间内连续改 reg2/reg3/reg0/reg9。回顾里程碑 1 的命门：蜂鸣器（Buzzer）每次输入变化都会置 updateRequired=true 并重建音频缓冲（见 ref/.../Buzzer.java 的 propagate() 和后台线程）。如果把高速变化直接喂进去，必然爆音/卡死。
保护办法是把送进蜂鸣器（Buzzer）的值按音乐帧率（约 60Hz）采样保持：无论上游变多快，蜂鸣器（Buzzer）每秒只被更新几十次。它是本里程碑从“手测”过渡到“自动化”的保护层；等接 CPU 时，最好把这根采样脉冲换成 CPU 的“本帧写完/进入 WAIT”提交信号，而不是永远用一个自由跑的慢时钟。

G1

给 Channel_Pulse 增加一个输入 frame_clk

回到 Channel_Pulse，加一个输入 引脚（Pin）frame_clk(1 位)。它是“每帧一拍”的慢时钟，由外面送进来（下一步生成）。

G2

在蜂鸣器（Buzzer）前插一排采样 寄存器（Register）

把原本“直连蜂鸣器（Buzzer）”的 4 路信号，各经过一个 寄存器（Register） 再进蜂鸣器（Buzzer）：
• FREQ：14 位 寄存器（Register）；  • VOL：4 位；  • PW：8 位；  • ENABLE：1 位。
每个采样 寄存器（Register）：D 接原信号、Q 接蜂鸣器（Buzzer）对应脚、时钟脚统一接 frame_clk、写使能接 1 位常量 1、清除脚接 1 位常量 0。于是蜂鸣器（Buzzer）只在 frame_clk 上升沿才看到新值。

至少把 FREQ 锁住

嫌 4 个寄存器多，最低限度给 FREQ 加采样保持即可（频率变化是爆音主因）。但 4 路都锁最稳，推荐全锁。

这一步在干嘛（先看这句再动手）

仿真时钟每秒跑上千拍，但蜂鸣器（Buzzer）每秒只该被刷新 ~60 次。所以要分频：让一个 计数器（Counter）数到 N 就回绕，把它的进位（Carry）当 frame_clk——进位每 N 拍才冒一个头，正好把快节拍压成 ~60Hz。照 G3 → G4 → G5 三步搭即可；为什么取 N=33、能不能正好 60、有没有更糙的替代招，都挪到本节末「选读」，不挡你搭。

G3

回 main：放 时钟（Clock） + 计数器（Counter），把时钟接进计数器

线路 → 时钟（Wiring → Clock） 放一个，属性全保持默认（高电平持续时间（High Duration）=1、低电平持续时间（Low Duration）=1，源码 Clock.java:181——即每 2 个 tick 走完一个完整周期）。
存储器 → 计数器（Memory → Counter） 放一个。先认清它这三个脚（鼠标悬停看 tooltip 最稳，源码 Counter.java:126/131）：

  • 时钟脚：在计数器左侧、画了 ▷ 三角、旁标 2,3,5+/C6 的那个（tooltip「输入：时钟」）。嫌难认就把 外观（Appearance） 换 经典（Classic），时钟脚是底部带 ▷ 的。
  • 进位（Carry）：右侧偏上那个 1 位输出，旁标 3CT=0x…（tooltip「输出：进位（递增到最大值…时为 1）」）。这就是我们要的 frame_clk 源。
  • Q：右侧偏下的多位总线（tooltip「输出：当前计数值」）——本步用不到，空着即可。

只接一根线：时钟（Clock） 的输出 → 计数器（Counter） 的时钟脚。左侧其余脚（R / 载入 / 增减 / 使能 / 数据 D）全部空着——悬空时计数器默认就是「使能 · 向上数 · 不清零 · 不加载」，会自己一直往上数（源码 Counter.java:540–542：这些脚 ≠ 0 即视为生效）。

偷懒法

你在 BuzzerStress 里已搭过一模一样的 时钟（Clock）+ 计数器（Counter）。可框选复制进 main 省得重放——本步与里程碑 1 唯一的差别是：那时取 Q 的高位当 ROM 地址，这次改用 进位（Carry）。

G4

把计数器设成「除以 33」，进位（Carry）→ frame_clk

选中 计数器（Counter），改这三个属性，让它每 33 拍回绕一次：

  • 数据位宽（Data Bits）=8（装得下最大值 32 即可，8 位最省心）。
  • 最大值（Maximum Value）= 20。⚠️ 这个框按十六进制解析：要「除以 33」就填 20（= 十进制 32 = N−1），千万别填 33（会被当成 0x33 = 51）。
  • 溢出时行为（Action On Overflow）= 回绕（Wrap around）。

于是它数 0,1,…,32,0,…，每 33 拍一轮；进位（Carry）在「数到 32」那一拍变 1（源码 Counter.java:582–583：向上计数时 进位 = 当前值到达最大值），所以进位每 33 拍冒一个上升沿。
把 计数器（Counter） 的 进位（Carry）脚 → main 里 Channel_Pulse 实例的 frame_clk 输入脚。采样率 = R / 33（R 见下一步）。

G5

开 自动节拍（Auto-Tick）、设 4kHz，重做 §4 验证

勾选 仿真 → 启用自动节拍（Simulate → Auto-Tick Enabled），把 仿真 → 自动节拍频率（Simulate → Auto-Tick Frequency） 设 4kHz。此时 Logisim 左上角显示的「完整周期/秒」≈ R = 2000（= 菜单频率 ÷ 2，源码 TickCounter.java），故 frame_clk ≈ 2000 / 33 ≈ 60.6Hz。
务必 >1000Hz：节拍 ≤1000Hz 时 Windows 定时器精度会把它卡在 ~32/秒（见 §4.5 顶部红框），frame_clk 跳不稳、采样 寄存器（Register） 不更新、蜂鸣器（Buzzer）就不出声。用 进位（Carry）÷N 时分频比 N=33 固定，把节拍调高只让刷新更快、不改 60.6 这个比例，所以 2kHz / 4kHz / 8kHz 都行（越高越稳）。
重做 §4 那四笔写寄存器：应当照常听到 A4，而且之后改 timer / 音量时干净不爆——这就是采样门生效了。
想确认 frame_clk 真在跳：在那根线上挂个 探针（Probe），运行时它会飞快闪 0/1（太快看不清细节，但若卡死在 0 或 1、或显示蓝色「未知」，就是时钟/计数器没接好）。

选读：为什么是「约 60」而不是正好 60 · 有没有更省事的招

为什么连「除以 N」也凑不到正好 60

左上角能取的 R 只能是 500 × 2^k（500 / 1000 / 2000 / 4000 …，因为 自动节拍频率（Auto-Tick Frequency）>1kHz 的档只有 1 / 2 / 4 / 8 / … kHz，源码 MenuSimulate.java）。这串数没有因子 3，而 60 = 2²·3·5 含一个 3。要 R / N = 60 就得 N = 25·2^k / 3 为整数 ⟹ 需 3 | 2^k ⟹ 永不成立。所以精确 60 数学上够不到，只能逼近：4kHz + N=33 → 60.6Hz（+1%） 已经很近；想更近用 8kHz + N=67 → 59.7Hz（−0.5%）（此时 数据位宽 ≥ 7、最大值 = 42 即 0x42=66）。

更糙但更省事的替代：不设最大值，直接取计数器某一位

嫌设 最大值 麻烦，可回到老办法：计数器属性全默认，放一个 Splitter（分线器）取它输出 Q 的第 n 位当 frame_clk（输入位宽（Bit Width In）= 计数器位宽、分支数（Fan Out）=2，把 Bit n 单独分到一支、其余 Bit 全丢到另一支不接）。采样率 ≈ R / 2^(n+1)，只能得到「2 的幂分之一」：R=1000（2kHz 节拍）取 Counter[3] 得 1000 / 16 = 62.5Hz——这就是 62.5 的来历，2 的幂分频里离 60 最近的一档。和里程碑 1 BuzzerStress 取高位扫表是同一招，只是这次取 1 根线。它比 进位（Carry）÷N 法差一截（62.5 vs 60.6），但少设两个属性。两种二选一，别同时接两个源到 frame_clk。

别为这 2~2.5Hz 较真：对采样门它基本是化妆品

两点诚实交代：① NES 真实帧率是 60.0988Hz，「60」本身也是约整。② 这个 frame_clk 只是蜂鸣器（Buzzer）的刷新率——60 还是 62.5 听不出区别，它只需「够快不漏帧、够慢不让蜂鸣器（Buzzer）抖」。更关键的是：接了 CPU 后它会被 frame_commit 取代（见下方提示框、里程碑 3 §5.3），那时采样脉冲与音乐同源、零漂移，根本不靠这个自由跑的分频。真正决定「节奏精不精确」的是 CPU 时钟 × frame_cycles，靠现场调 自动节拍频率 校准。所以把它做成精确 60 是锦上添花 + 顺便演示 最大值/进位，并非必须。

接了 CPU 之后这道门会怎样

下一里程碑 CPU 用 WAIT 让 PC 停顿，把写寄存器的节奏天然压到音乐帧率（~60Hz）。但同一帧开头可能连续写好几个寄存器（比如先写 timer 低位、再写高位、再写音量），所以自由跑的 frame_clk 有概率采到“半更新”的组合。最终接 CPU 时，最佳做法是让 CPU 在一帧写完、开始执行 WAIT 时给 APU 一个 frame_commit 脉冲，并把它接到这里的 frame_clk。在本里程碑手测阶段，先用上面的分频慢时钟即可。

P2

Pulse2：优先用「同一个 Channel_Pulse 的第二个实例」

Pulse1 和 Pulse2 的电路完全一样，只是喂不同寄存器。Logisim 里一个子电路定义可以被放任意多个实例（就像画布上放两个与门）——所以最干净的做法是不新建电路：等 P5 封 APU 时，把 Channel_Pulse 从左侧列表再拖一次到画布（这是「第 2 个实例」，不是「第 2 个电路」），输入 reg0/reg2/reg3 改喂 reg4_out/reg6_out/reg7_out、enable 喂 reg9 第 1 位即可。

「子电路不能同名」≠ 不能放两个方波

那条限制只针对电路定义的名字（不能有两个都叫 Channel_Pulse 的定义）；同一个定义放多少个实例都行，不算重名。
当然，复制出一个独立的 Channel_Pulse2（改了名的副本）当 Pulse2 也完全能用，只是多一份维护：以后改 Channel_Pulse（比如调采样门 / 占空比）必须同步改 Channel_Pulse2，别让两个方波行为不一致。已经建了 Channel_Pulse2 就继续用，保持两边一致即可。

同样复制改名 Channel_Noise。它最特别：频率直接拿 reg14（=$400E）低 4 位当 ROM 地址，不做 11 位 timer 拼接（$400E 低 4 位 = NES 噪声「周期索引」，共 16 档）。改动如下：

★ 源码修正（2026-06-09）：蜂鸣器噪声"有音高"，freq_noise 必须用低频

蜂鸣器（Buzzer）噪声波形虽是逐采样随机（Buzzer.java:300 Math.random()*2−1），但缓冲构建（Buzzer.java:358/381-383）把一段长度 = sampleRate/hz 的随机块按 hz 周期性平铺——所以输出是"基频 = hz 的带噪嗡声"，hz 越高越像尖锐纯音。所以噪声有音高、受 reg14→freq_noise 影响（本文档早期"无视 hz / 改 reg14 音色不变 / 可省掉 freq_noise ROM"是错的，已作废）。
关键后果：freq_noise 绝不能填 NES 的 LFSR 时钟频率（CPU/period，几百~几万 Hz，会刺耳尖啸）。make_tables.py 已改成把 16 档映射到 40~200Hz（默认，可调） 低频区间（对数等分、index 0 最亮）。务必重新 加载映像（Load Image…） 新的 script/out/freq_noise.txt 到噪声通道 ROM。还嫌尖 → 把脚本里 NOISE_HZ_MAX 调更低（如 250）再重生成。

单测噪声通道

测试台写 reg9=0x08（只开 bit 3 = 噪声）、reg12=0x0F（满音量）→ 应听到"沙沙"带噪嗡声（鼓 / 镲那种，低频不刺耳=新表生效）；reg12=0x00 几乎无声；reg9=0x00 整体静音。改 reg14 低 4 位会改变音高/音色（不同鼓不同低频嗡声）——这是对的；若改 reg14 听到的是尖锐高音，就是还在用旧的高频 freq_noise，重载新表即可。

先讲清 隧道（Tunnel）怎么用 —— P5 全靠它接线

P5 要把 RegFile 十来个输出、frame_clk、reg9 各位…接到 4 个通道，直接拉线必乱成一团。隧道（Tunnel）= 按名字连线：同一个电路里所有同名（且同位宽）的隧道，等价于用导线连在一起，不用真把线拉过去。

  • 长什么样：一个带尖角的小「旗子」（五边形），旗面写着名字；尖角那一点是它唯一的连接点（源码 Tunnel.java:109：单个 INOUT 端口、就在锚点处）。尖角朝向由 朝向（Facing） 决定。

两个隐形坑（P5「没声 / 某路不响」十之八九是这俩）

5.5★ 用平铺 ROM 试听真实曲目（免 CPU）

为什么现在就能听整段真曲（还没搭 CPU）

手写寄存器只能听单音/和弦。要让 APU 自动按谱发声，本来需要 CPU 去「解释」指令流里的 WAIT（等 n 拍）。捷径：在离线侧把每条 WAIT n 展开成 n 行 NOP，做一条「平铺 ROM」——节奏被烤进 ROM 的行数里，于是一个傻瓜计数器（Counter）逐行扫描就是正确的实时播放，完全不需要 WAIT 状态机。这既能提前验证 APU 在真实寄存器流量下的听感（detailed_plan §8 最担心的 Buzzer 性能风险），又正好是「里程碑 3 减掉 WAIT 状态机」的踏脚石。

先分清：平铺 ROM 的「地址位宽」≠ 成品能装多长的曲子

这台是临时试听脚手架，不是成品。平铺 ROM 把 WAIT 摊成一行行 NOP，极费空间：每秒约 1920 字（60 帧 × frame_cycles 32）。所以这里的地址位宽只决定「能听多长的片段」：15 位 = 32768 字 ≈ 17 秒，16 位 = 65536 字 ≈ 34 秒。整首 4 分钟摊平要 ≈46 万字（需 219 地址），故意不做——平铺台只听个开头。
最终成品根本不用平铺 ROM。它用 CPU 读带 WAIT 的压缩指令 ROM（track00..15.txt，地址 16 位 / 数据 16 位），WAIT 由 CPU 解释让 PC 停顿，所以整首装得下：实测最大 track11 = 41044 字 = 216 的 62.6%，16 首全部 < 65536。「把 WAIT 压回去」正是做 CPU（里程碑 3）的意义，平铺台只是免 CPU 先听个响。

位宽要对齐：生成位宽 = ROM 地址位宽 = 扫描计数器位宽

加载时 ROM 的 地址位宽（Address Bit Width） 必须＝你生成时填的那个位宽（…15 → 15、…16 → 16），且 A2 的扫描 计数器（Counter） 数据位宽（Data Bits） 也要设成同一个数。三者不一致就会出现「decoded 65536 of 32768 words」这类装不下 / 只播一半的警告——例如把 16 位生成的 flat_demo_ff3.txt（65536 字）塞进 15 位 ROM。

Splitter（分线器）：输入位宽（Bit Width In）=4、分支数（Fan Out）=4（每位一支）。把它的合并端（那根 4 位侧）接到 APU 实例的 ch_mask 输入脚。再把 拨码开关（DIP Switch）的 4 路输出按位接到 4 个 1 位分支：

第 1 路分支 0bit 0Pulse1
第 2 路分支 1bit 1Pulse2
第 3 路分支 2bit 2Triangle
第 4 路分支 3bit 3Noise
哪一格对应哪一位？用「拨一格、听哪路静音」来核对

本版 拨码开关（DIP Switch）每格的悬停提示只回退显示成 DIP（源码缺逐格文案 key），看 tooltip 认不出第几格。所以哪格 = 哪位，以位置顺序为准；最稳的是开声后只拨一格、听哪一路静音来标定（或在合并器每个分支上挂一个 探针（Probe） 看 0/1）。把对应关系记到上表即可。

5.6.3 为什么旧测试台一行都不用改（上拉安全网）

§4 / §4.5 / §5「手动测 4 路」/ §5.5 试听台这些地方实例化 APU 时都没接 ch_mask。但因为它设了 行为=上拉（Pull Up）：端口悬空 → 内部读全 X → 上拉拉成全 1 → ch_mask=0xF → en = reg9_out 原值。于是旧测试台照常四路齐响，无需任何改动；只有当你真的接上 DIP 开关时，开关才接管。这就是选「上拉」而不是「下拉/简单」的原因——默认安全（全开），接了才管。
反例提醒：若你把它设成 简单（Simple）或忘了上拉，悬空时 ch_mask 是全 X，与门一与 → 想响的位变 X/红线 → 通道反而不出声。所以这一步别省。

5.6.4 层级提醒：开关在 top，口开在 APU——中间每多一层就透传一层

ch_mask 这个口开在 APU 上。top 的开关怎么够到它，取决于你的总装层级：

  • 推荐：top.circ 直接实例化 APU（CPU 与 APU 在 top 里并列、由 top 连线）→ DIP → 合并器 → APU.ch_mask 一步到位，最省事。
  • 若 APU 嵌在 cpu.circ 里、再由 top 实例化 cpu → 那 cpu.circ 也要照样多暴露一个 ch_mask(4) 输入脚做透传（cpu 内部把它直接连到内部 APU 实例的 ch_mask），top 的开关才接得上。每多一层中转电路，就照此多透传一层同名口。
5.6.5 现在就能测 + 答辩玩法

DIP 开关属于 top.circ 控制面板，里程碑 4 才正式组装；本里程碑只要求把 ch_mask 这个口在 APU 上留出来、内部 与门（AND Gate）接好。但你现在就能在 §5/§5.5 的测试台上验证：在测试台放一个 4 路 拨码开关（DIP Switch）+ 合并器，接到 APU 实例的 ch_mask，一边播平铺 ROM（§5.5）一边拨开关——能实时静音 / 恢复任意一路就算通。

答辩演示三招：① 独奏某路 = 只留它的开关开、其余三个关（「这是方波主旋律」「这是三角波低音」「这是噪声鼓点」，一路路点给评委听）；② 关掉某路 → 听感立刻少一层，直观证明四路是各自独立的硬件通道；③ 全部关 = 整体静音的另一种演示（与写 reg9=0 等效，但不改歌曲数据、纯硬件开关，更好讲）。

★ 答辩必备：蜂鸣器（Buzzer）的两条音质边界 + 怎么说 + 怎么治

本项目用 Logisim 蜂鸣器（Buzzer）当音源，有两条源码层面的固有限制。答辩前务必能主动解释（主动讲＝加分，被问住＝减分）。

① 长音「爆破 / 断续」——蜂鸣器每次输入变化都重建整段音频。

源码 Buzzer.java：任一输入变化 → propagate 置 updateRequired=true（254 行）；后台线程随即关掉旧 Clip、重算缓冲、新开一个 Clip 重放（401–412 行）。hz != oldfreq（350 行）只管要不要重算采样率，Clip 本身是每次都重建——所以 FREQ / VOL / PW / ENABLE 任何一个变一下，就重开一次 = 一声「咔」。

  • 真·恒定长音（频率/音量都不变）→ 根本不触发 propagate → 不重建 → 平滑。
  • 带每帧调制的音（如 FF3 的音量包络 envelope + 颤音 vibrato，每帧改 VOL/FREQ）→ 每帧重开一次 → ~60Hz 连续「爆破」，听起来像断续而非长音。Monitoring 音更稳、调制少，所以顺（它"快"只是顺带）。

对策（按需取舍）：

  • 硬件侧：§4.5 的按帧采样把重建率封顶在 ~60Hz；接 CPU 后改用 frame_commit（一帧写完后才采一次）→ 消掉"帧内半更新"造成的额外爆破 + 对齐节奏 + 取代傻瓜扫盘。能明显改善，但治不了"每帧确实在变"的那 60Hz 底噪。
  • 软件侧（真正的解药，离线在 encode.py 调）：① 音量包络拍平（一个音内 VOL 保持不变）；② 频率死区（小于阈值的颤音抖动忽略）；③ 限更新率（每 N 帧才允许改一次）。拿表现力（颤音/渐弱）换平滑，逐曲可调。

② 噪声音色 ≠ 原曲——蜂鸣器噪声是"有音高的带噪嗡声"，且不能喂 NES 的 LFSR 频率。

蜂鸣器噪声虽逐采样随机（Buzzer.java:300），但缓冲把随机块按 hz 周期平铺（358/381-383）⇒ 输出是基频 = hz 的带噪嗡声，hz 越高越尖锐。所以 freq_noise 必须映射到低频（本项目 40~200Hz（默认，可调），make_tables.py），否则刺耳（NES 的 LFSR 时钟有几万 Hz，直接用就是尖啸——这正是先前 freq_noise 用 CPU/period 导致"噪声异常尖锐"的根因，已修）。即便如此，蜂鸣器也做不出 NES 噪声真正的宽带"沙沙"质感和金属短模式——只能给不同鼓不同的低频嗡声音高，是近似。另：部分曲子的鼓用 NES 的 DMC 采样通道，按锁定决策已砍（$4010-4013），打击层会再缺一块。
