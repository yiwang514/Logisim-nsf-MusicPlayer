02转换工具链（已完成，供评审了解输入端）

阶段一：为什么用 GME 而非 NSFPlay

GME 的 Nes_Apu::write_register(time,addr,data) 是处理全部 $4000-$4017 写入的单一函数，自带 time 参数；NSFPlay 把写寄存器拆在两个文件、无时间戳。对 ref/game-music-emu 做了 3 处 #ifdef GME_APU_DUMP 保护的最小改动（详见 script/gme_dump/PATCH.md）。构建注意：不可用 NSF_EMU_APU_ONLY（MinGW 下堆越界）、emu2413.c 须用 gcc 单独编、需 -DBLARGG_LITTLE_ENDIAN=1 与静态链接。

Buzzer 的 Frequency 引脚按真实墙上时间发声（与 Logisim 仿真快慢无关），所以频率表输出真实 NES Hz，不乘任何“Logisim/NES 时钟比”。只有节奏（WAIT 时长）受仿真时钟影响，靠 CPU 时钟频率 + encode.py --ticks-per-frame 校准。这一点修正了原计划书 S5 的“频率×缩放系数”（那是针对自搓振荡器的）。

03指令集架构 ISA（16-bit 定长）

统一 16-bit 等长指令，高 4 位 opcode，低 12 位操作数。定长 = 取指就是 PC→ROM→PC+1，无变长状态机，硬件最简。指令 ROM 字地址、数据 16-bit。

op助记位布局 [15..0]语义字数
0x0WAIT0000 | imm12等待 imm12 个时间单位（0–4095）1
0x1WRITE0001 | reg(4) | val(8)APU[reg] = 立即数 val1
0x2LWAIT0010 | ──── ; 次字=16-bit 等待数长等待（>4095 时用；本编码器几乎不产生）2
0x3WKEY0011 | reg(4) | val(8) ; 次字=MIDI(8)|──同 WRITE + 附 MIDI 音符（可视化用，可选）2
0x4LOADI0100 | Rd(2)|── | imm8R[Rd] = 立即数（ALU 路径）1
0x5ADD0101 | Rd(2)|Ra(2)|Rb(2)|──R[Rd]=R[Ra]+R[Rb]（ALU）1
0x6SUB0110 | Rd(2)|Ra(2)|Rb(2)|──R[Rd]=R[Ra]−R[Rb]（ALU）1
0x7WRITEREG0111 | reg(4) | Rs(2)|──APU[reg] = R[Rs]（把算出来的值写进 APU）1
0xFEND1111 | ────曲目结束 / 循环点（PC 归零）1
ALU 是老师明确要求（不可砍）

老师要求“利用 Logisim 搭建 CPU，可自由发挥，建议实现可控加减运算”。因此 LOADI/ADD/SUB/WRITEREG + 4 个通用寄存器 R0–R3 + 真 ALU 是核心交付物。注意：自动回放的 ROM 只用到 WAIT/WRITE/END 三条；ALU 指令通过两种方式被执行/展示：
① 一段手写 demo ROM（LOADI R0,100 / LOADI R1,50 / ADD R2,R0,R1 → R2=150，现场可视化）；
② 可选：在 encode.py 里加颤音检测，把“连续 ±1 改频率”的模式编码成 LOADI + ADD/SUB + WRITEREG，让真实音乐里也跑 ALU。

WRITE 的 reg(4) → APU 寄存器映射（APU 译码器据此解码）

reg_id归属原 NES 地址
0–3Pulse 1$4000–$4003
4–7Pulse 2$4004–$4007
8, 10, 11Triangle$4008,$400A,$400B
9声道使能（重映射）$4015（低 4 位 = N/T/P2/P1）
12, 14, 15Noise$400C,$400E,$400F

$4015 落在 4-bit reg_id（只能 $4000–$400F）之外，借真实 NES 未用的槽 reg 9（=$4009）承载；真实 $4009/$400D 写入被编码器丢弃以避免冲突，$4017 丢弃，DMC $4010-$4013 在 dumper 侧已跳过。

05CPU 详细设计（单周期五段 + ALU）

单周期、五个组合阶段 IF/ID/EX/MEM/WB（非流水线，见 §8 的需求歧义警告）。组件量估计 70–90。

数据通路

IF : PC ─► 指令ROM(MUX 选当前曲) ─► instr(16)
ID : opcode=instr[15:12] ─► 控制信号；拆 reg/val/imm/Rd/Ra/Rb
EX : ┌ WAIT  → 装载/递减 WAIT 计数器，产生 stall
     ├ ALU   → ADD/SUB(8-bit)，结果 → R[Rd]（LOADI 直接载入）
     └ 地址  → reg_id
MEM: ┌ WRITE/WKEY  → APU[reg_id] = val(立即数)      （we = is_write && !stall）
     ├ WRITEREG    → APU[reg_id] = R[Rs]            （数据来源 MUX 选择）
     └ END         → 触发“曲目结束”→ PC=0（循环）
WB : PC 更新：stall ? 保持 : (END ? 0 : PC+1)；ALU 结果写回 R[Rd]
★ 唯一真正的正确性陷阱：WAIT 的 stall 重触发

WAIT 用“PC 原地不动 + 计数器每周期 −1”实现。陷阱：计数器归零那刻，PC 上还是那条 WAIT，若逻辑是“看到 WAIT 就装载计数器”，下周期会再次装满 → 死循环。
推荐解法（busy 标志状态机）：加 1-bit wait_busy 寄存器。
  • busy==0 且 instr==WAIT → 装载计数器 = imm，置 busy=1，PC 不动；
  • busy==1 → 计数器 −1；当减到 0 → busy=0 且这一拍 PC+1；
  • 其它指令 → 单周期执行，PC+1。
本质是给单周期 CPU 加一个最小的“多周期指令”状态机。这是必须在纸面审清的一处。

ALU 子电路

一个 8-bit 加法器 + 一个 8-bit 减法器 + 一个按 op 选结果的 MUX。LOADI 走旁路直接把立即数送 R[Rd]。验收演示：主电路放 4 个 Hex Digit 显示 R0–R3，现场改 demo ROM 跑加减。

06APU 详细设计（4 路 Buzzer 并联）

APU = 把 CPU 写进来的寄存器值 → 算频率/音量/使能 → 喂 Buzzer。不写波形、不写混音（Buzzer + 系统声卡搞定）。每通道结构一致：寄存器 → 频率换算 ROM → Buzzer。

通道Buzzer 波形频率来源音量 / 占空比
Pulse 1 / 2Square11-bit timer → freq_pulse ROMvol=$4000/$4004 低4位；duty=$4000 bit7-6 → {32,64,128,192}
TriangleTriangle11-bit timer → freq_triangle ROM无音量寄存器 → VOL 接常量；使能控响不响
NoiseNoise$400E 低4位 → freq_noise ROM（16项）vol=$400C 低4位

$4015 使能：reg 9 低 4 位分别 AND 到 4 个 Buzzer 的 ENABLE。停止/暂停时把 reg 9 写 0（或门控）即可瞬间静音。4 个 Buzzer 各开一条音频 Clip，系统声卡自动叠加（已核实，见 §8）。

07控制面板 / 多曲切换 / 计时

  • 播放/暂停：暂停标志 D 触发器门控 CPU 时钟（冻结状态）。
  • 停止：复位 PC=0 + 写 reg9=0 静音。
  • 上一首/下一首：4-bit 曲目计数器 → 16:1 MUX 选哪个指令 ROM 喂 CPU。每首一个独立 ROM（track00..15.txt），不合并文件。切歌时复位 PC。
  • 自动下一首：计时器到上限（DIP 设 4/5/8 分钟）→ 边沿 → 曲目计数器 +1。
  • 计时器：1Hz 慢时钟 → 秒(0–59)→分，显示 MM:SS；切歌/停止复位。
  • END 信号：CPU 读到 0xF000 → PC 归零，无缝循环当前曲。

08可行性审计 & 风险

风险 1 Buzzer 在“频率快速变化”时可能卡顿 —— 须 D1 实测

核对 ref/.../io/extra/Buzzer.java：
• 每次输入引脚变化触发 propagate，置 updateRequired=true（254 行），后台线程随即重建整整 1 秒音频缓冲（byte[4*sampleRate] ≈176KB）并重开一个 Clip（386、401 行）。
• 开发者注释自警：把 4kHz 时钟接 enable 会崩（432 行），并有 Thread.activeCount()>100 保护（433 行）。
含义：长音（参数不变）零开销；但 FF3 这类每帧改频率的颤音会让线程每秒重建几十次缓冲 × 4 路。约 60–240 次/秒，远低于 4kHz 警戒线，大概率可用但可能爆音/卡顿。这正是“单音 OK、真曲子垮”的典型陷阱，必须先测。
缓解（若卡）：① 在 Buzzer 输入前加一组采样寄存器，用 ~30–60Hz 慢时钟“按帧采样”再喂（计划书 S9 给可视化用的思路，同样用于声音）；② 在 encode.py 加节流，限制频率更新率（牺牲颤音换流畅）。

已确认 “五段 CPU” 不要求流水线

老师已明确：做到 IF/ID/EX/MEM/WB 五个阶段即可，不要求流水线（不需处理数据/控制冒险）。本计划的单周期五段方案满足要求，不存在因此推倒重来的风险。（此处原为高危需求歧义，现已澄清并降级。）

风险 2 节奏等时：每帧耗时须恒定

当前 encode.py 每帧只补 WAIT 1，但一帧内 WRITE 条数不等（3–20+）→ 每帧实际周期数不等 → 节奏忽快忽慢。
修法：令 WAIT = 每帧固定预算周期 − 本帧写入条数（预算取 ≥ 最大单帧写入数，如 32），使每帧恒为固定周期。属编码器小改，已识别、待实施。

已核实可行的点
  • Buzzer 引脚（FREQ14/ENABLE1/VOL7/PW8）与波形（Square/Triangle/Noise）齐全，与设计吻合。
  • 频率表全部落在 20–20000Hz 内（Pulse≥55、Tri≥27、Noise≥440），不会触发静音。
  • 多 Buzzer 各自 AudioSystem.getClip() 独立播放，由系统混音 → 4 路并联成立。
  • ROM/PC/译码都是标准数字逻辑，无特殊依赖；16 首 + 频率表已校验为合法 v2.0 raw。
  • 位宽全表自洽（§4）；PC 位宽是可后调属性，非地基。

10对原计划书 ref/project_brief_v6.html 的勘误

  1. PC 12-bit → 16-bit：4 分钟曲子最多 41043 条指令 > 4096，12 位会溢出回绕。
  2. $4015 → reg 9 重映射：4-bit reg_id 装不下 $4015，借未用槽 reg 9。
  3. 音高不缩放：Buzzer 用真实 Hz；S5 的“频率寄存器值 × 时间缩放系数”仅适用于自搓振荡器，本方案不用。
  4. 节奏等时：每帧 WAIT 应补到固定周期预算，而非恒为 1（见 §8 风险 2）。
  5. hook 目标改 GME：原 S5 写 NSFPlay，实际用 game-music-emu（单一 hook 点 + 自带时间戳，构建更顺）。
