  • apu.circ 里已有可用的 APU 顶层子电路：对外是写口 reg_id(4) / value(8) / WR(1) + frame_clk(1)，内部 RegFile（输出脚 regN_out）+ 4 个通道 + 蜂鸣器（Buzzer）。手写寄存器能出 A4、能出和弦、能整体静音。

0这一里程碑要搭什么

从「手写寄存器」到「CPU 自己读谱」

里程碑 2 里你扮演 CPU：拨 reg_id/value、点 WR。本里程碑把这个「手」换成一颗 CPU——它每个时钟拍取一条 16 位指令，看高 4 位 opcode 决定干什么：WRITE 就把操作数送到 APU 写口并打一个写脉冲；WAIT 就原地停顿若干拍（这就是音乐的「时值/休止」）；END 就把 PC 归零、整曲循环。指令流来自 track00.txt，是离线工具链早就生成好的。

指令格式回顾（16 位定长，高 4 位 opcode；本里程碑只需这 3 条）
opcode助记位布局 [15..0]CPU 要做的事track00 例
0x0WAIT0000 | imm12原地停顿 imm 个时钟拍（PC 不动）0020=WAIT 32（满一空帧）、001f=WAIT 31
0x1WRITE0001 | reg4 | val8APU[reg]=val，并打一个写脉冲190f=写 reg9=0x0f（开声道）
0xFEND1111 | ────PC 归零，循环本曲f000（在 ROM 末尾）

0x2 LWAIT（长等待、2 字）等其余 opcode 本曲用不到，留给里程碑 4。回放永远只用 WAIT/WRITE/END；LOADI/ADD/SUB/WRITEREG 是 ALU 指令，里程碑 4 才登场。

整体数据通路（五段，单周期、非流水线）

老师确认「五段」= 做到 IF/ID/EX/MEM/WB 五个组合阶段即可，不要求流水线（无需处理冒险）。所以这是一颗单周期机器：一条指令在一个时钟拍内走完五段（WAIT 例外，它要停顿多拍——这正是 §4 的状态机）。

IF   PC(16) ─► track00 ROM (addr16 / data16) ─► instr(16)
ID   分线器拆位:  opcode=instr[15:12]   reg_id=instr[11:8]   val=instr[7:0]   imm=instr[11:0]
     译码器(4→16): 第0路=is_wait   第1路=is_write   第15路=is_end
EX   WAIT 状态机 = busy 触发器 + 16 位"向下计数器"  →  产生 stall（PC 停不停的总开关，见 §4）
MEM  WRITE: reg_id / value 送 APU 写口,  WR = is_write 与 时钟 相与（写脉冲）
     END  : PC 装 0
WB   PC 更新 = stall ? 保持 : (is_end ? 0 : PC+1)   ←── 用一个"向上计数器"天然实现，无需额外加法器

输出到 APU(里程碑 2):  reg_id(4)=instr[11:8]   value(8)=instr[7:0]   WR=is_write·clk   frame_commit=start·clk
为什么用「计数器」而不是「寄存器 + 加法器」做 PC

PC 的行为正好是 stall?保持 : END?0 : +1。Logisim 的 计数器（Counter）原生就有：计数使能（不使能=保持）、加载（装 0）、向上计数（+1）三件事。所以 PC = 一个 16 位 计数器（Counter） 就够了，省掉「寄存器 + 加法器 + 多路复用器」三件套。同理 WAIT 的倒计时也用一个向下计数器。这两个计数器是本里程碑的主角，§1 先把它的引脚按源码讲清。

Counter.java 的 propagate()（536–595 行）确认两件事，本里程碑直接吃红利：

  • 进位（Carry）是组合输出、且向下计数时 = 「当前值是否为 0」：carry = (newVal == goal)，向下计数 goal=0。它不等时钟、随当前值实时变化。⟹ 不用再放比较器（Comparator）去判 0，WAIT 计数器的 进位 脚就是现成的 at_zero 信号。
  • 加载优先于计数（550 行 else if (ld) 在计数分支之前）：同一拍 Load=1 时只装载、不计数。⟹ 装入 imm 的那一拍不会同时又减 1，时序干净。

另：清除（Clear）是异步的（536 行在时钟判定之外），所以 reset 一按立刻清零；计数使能/方向悬空都按"真"处理（541–542 行 != FALSE），所以这两个脚必须明确接常量，否则行为不是你以为的那样。

怎么在画布上认计数器（Counter）的 8 个脚

默认 logisim_evolution 外观下，符号左侧从上到下标着控制脚：R=清除、M1[load]=加载、M3[up]/M4[down]=方向、G5=计数使能、最下带三角(2,3,5+/C6)=时钟；数据 D 是左下那条宽总线(标 1,6D)、计数值 Q 是右下宽总线、进位 Carry 在右侧(标 CT=0)。（这些标注来自源码 Counter.java 的 drawControl()。）嫌符号大就把 外观（Appearance）=经典（Classic）：小方块，D 左、Q 右、时钟在底(带三角)、其余控制脚贴边。不管哪种外观，鼠标悬停弹出的中文 tooltip 才是认脚的硬依据，别凭方位猜。

1.2 译码器（Decoder）—— opcode 4 位 → 16 根 one-hot 控制线

库：复用器 → 译码器（Plexers → Decoder）（里程碑 2 的 RegFile 已用过）。设 选择位数（Select Bits）=4 ⟹ 16 路输出，选中那路=1、其余=0。本里程碑把 opcode 接到它的选择脚，只取第 0 路（is_wait）、第 1 路（is_write）、第 15 路（is_end），其余悬空。
若它带使能脚（属性 启用使能（Include Enable?））：把该属性设否，或给使能脚接 值=1 的 常量（Constant），否则整片输出全 0。

1.3 逻辑门（Gates）—— 拼 WAIT 状态机的几个控制信号

库：逻辑门（Gates）。本里程碑只用三种、全是 1 位：与门（AND Gate）、或门（OR Gate）、非门（NOT Gate）。用途见 §4：start = is_wait 与 ¬busy、en_count = busy 与 ¬at_zero、busy_next = start 或 en_count，外加几个 非门 取反。

1.4 位扩展器（Bit Extender）—— 把 12 位 imm 补成 16 位

库：线路 → 位扩展器（Wiring → Bit Extender）（源码 std/wiring/BitExtender.java）。属性：输入位宽（Bit Width In）=12、输出位宽（Bit Width Out）=16、扩展类型（Extension Type）=全 0（Zero）。把 imm(12) 接它的输入，输出就是高 4 位补 0 的 imm(16)，喂给 WAIT 计数器的 数据 脚。
不想加这个部件也行：用一个 Splitter（分线器）当合并器——imm(12) 进低 12 位、一个 值=0 的 4 位 常量（Constant）进高 4 位，合并成 16 位，效果相同。

3译码：分线器拆位 + 译码器出控制信号

把 16 位 INSTR 拆成 opcode / reg_id / val / imm，再用译码器把 opcode 变成「这是不是 WAIT / WRITE / END」的 one-hot 信号。

D1

第一个 Splitter（分线器）：拆出 opcode 和低 12 位

放一个 Splitter（分线器）：输入位宽（Bit Width In）=16、分支数（Fan Out）=2。位分配：Bit 0–11 → 分支 0（这就是 imm，也是 reg+val 的合并），Bit 12–15 → 分支 1（这就是 opcode，4 位）。INSTR 接合并端。
分支 1 引出 → 隧道（Tunnel）OPCODE(4)；分支 0 引出 → 隧道（Tunnel）IMM(12)。

D2

第二个 Splitter（分线器）：把低 12 位再拆成 reg_id 和 val

再放一个 Splitter（分线器）：输入位宽=12、分支数=2。Bit 0–7 → 分支 0（val，8 位）、Bit 8–11 → 分支 1（reg_id，4 位）。把 IMM(12) 接合并端。
分支 0 → 隧道（Tunnel）VAL(8)；分支 1 → 隧道（Tunnel）REGID(4)。（reg_id/val 只有 WRITE 用得到；WAIT 用的是整段 IMM。同一组低 12 位两种解读，互不影响。）

D3

译码器（Decoder）：opcode → is_wait / is_write / is_end

复用器 → 译码器（Plexers → Decoder），选择位数（Select Bits）=4。把 OPCODE(4) 接选择脚。若有使能脚，属性 启用使能（Include Enable?）=否，或给它接 值=1 的 常量（Constant）。
取三路输出贴隧道（Tunnel）：第 0 路 → IS_WAIT、第 1 路 → IS_WRITE、第 15 路 → IS_END。其余 13 路悬空不接。

4★ WAIT 停顿状态机 —— 本里程碑的命门

陷阱本体：为什么不能「看到 WAIT 就装计数器」

WAIT 要让 PC 原地不动、同时一个计数器每拍 −1，减到头再放行。但 WAIT 那条指令在停顿期间一直躺在 PC 上（PC 没动），所以 IS_WAIT 会一直为 1。如果逻辑是「只要 IS_WAIT=1 就把计数器装成 imm」，那计数器每一拍都被重新装满 ⟹ 永远减不到 0 ⟹ 死循环、整曲卡在第一条 WAIT。这是 detailed_plan.html §5 点名的唯一真正正确性陷阱。

解法 = 一个 1 位 busy 标志，记住「这条 WAIT 我已经装过计数器了，现在是在倒数、不是刚遇到」。装载只在「刚遇到」那一拍发生，之后只倒数、不再装。

4.1 把状态机写成 5 个信号（先看清逻辑，再连线）

设 busy 是一个 1 位寄存器（触发器）的当前值，at_zero = WAIT 计数器的 进位（Carry）（=「当前计数值为 0」）。则：

is_wait   = 译码器第 0 路              （当前指令是不是 WAIT）
at_zero   = WAIT计数器.进位(Carry)     （倒计时是否已到 0，组合输出，源码已验证）

start     = is_wait  AND  (NOT busy)   （"刚遇到这条 WAIT"：装载就发生在这一拍）
en_count  = busy     AND  (NOT at_zero)（"正在倒数且还没到 0"：这一拍要 −1）

busy_next = start OR en_count          （下一拍 busy 的值 = busy 触发器的 D 输入）
stall     = busy_next                  （★ 二者恒等！见下方推导。PC 停不停就看它）
★ 关键化简：stall 和 busy_next 是同一个信号

PC 该停当且仅当：①这一拍刚遇到 WAIT（要开始等）= start，或②正在倒数还没到 0 = en_count。于是 stall = start OR en_count，和 busy_next 字面相同。所以你只需算一根线：busy_next。它同时当 busy 触发器的 D、又取反后当 PC 的 计数使能。少接一堆门。

4.2 连线（三个部件：WAIT 计数器、busy 触发器、几个门）

W1

放 WAIT 计数器（Counter），16 位，向下、到 0 保持

存储器 → 计数器（Memory → Counter），数据位宽=16，溢出时行为（Action On Overflow）=保持不变（Stay at value），标签 WAITCNT。接脚：

  • 数据（D）：把 IMM(12) 经 §1.4 的 位扩展器（Bit Extender，12→16，全 0） 补成 16 位再接进来。
  • 方向（Up/Down）：值=0 的 1 位 常量（Constant）（向下 −1）。
  • 加载（Load）：接 start。
  • 计数使能（Enable）：接 en_count。
  • 时钟：接 clk；清除（Clear）：接 reset。
  • 进位（Carry）：引出 → 隧道（Tunnel）AT_ZERO。计数值 Q 可不接（或接个 探针（Probe） 观察倒数）。
W2

放 busy 触发器（用 1 位 寄存器（Register））

存储器 → 寄存器（Memory → Register），数据位宽=1，标签 busy。D 接 busy_next、Q 引出 → 隧道（Tunnel）BUSY、时钟接 clk、写使能（en）接 值=1 的 1 位 常量（Constant）、清除（clr）接 reset。
（也可用 存储器 → D 触发器（Memory → D Flip-Flop）；1 位 寄存器（Register）和里程碑 2 一脉相承，这里沿用。）

W3

用逻辑门拼出 4 个信号

全部 1 位。建议用 隧道（Tunnel）取 IS_WAIT / BUSY / AT_ZERO 当输入，省得拉长线：

  • 非门（NOT）：BUSY → NBUSY；  AT_ZERO → NATZERO。
  • 与门（AND）：IS_WAIT, NBUSY → start（隧道 START）。
  • 与门（AND）：BUSY, NATZERO → en_count（隧道 ENCOUNT）。
  • 或门（OR）：START, ENCOUNT → busy_next（隧道 BUSYNEXT）。把 BUSYNEXT 接回 W2 的 busy 触发器 D。
  • 非门（NOT）：BUSYNEXT → NSTALL（= ¬stall）。

最后把 PC 计数器的 计数使能（Enable）由 §2 的临时常量改接 NSTALL。这样：停顿时 PC 不走，倒数完那拍 stall 落 0、PC 才 +1。

关于精确拍数：WAIT n 实际占用 n+2 拍（诚实交代，且不影响调速）

如上：装载占 1 拍（c0）、归零观察占 1 拍（c4），中间倒数 n 拍，合计 n+2 拍。这不是 bug，是这种「装载 + 倒数 + 判零」结构的固有常数。为什么不影响节奏：编码器让每一帧恰好一条 WAIT（WAIT = 32 − 本帧写入数），所以这 +2 常数每帧均匀地多 2 拍，整体相当于帧预算从 32 变 34——而真正的播放速度本来就靠 仿真 → 自动节拍频率 现场调（里程碑 2 的表），把节拍调高一点点就完全吸收了。
若日后想要精确 n 拍：把装入值从 imm 改成 imm−2 即可——而里程碑 4 的 ALU 本来就要在这条装载路径上算 imm ± tempo_delta（变速播放），那时顺手减常数、做下限保护（钳到最小 1），一举两得。本里程碑先按 imm 直装，结构清楚。

5WRITE 写脉冲 + END 循环

5.1 WRITE：给 APU 一个干净的「写脉冲」

为什么不能把 IS_WRITE 直接当 WR 接给 APU

里程碑 2 的 APU 写口里，WR 当的是寄存器的时钟，靠上升沿把 value 锁进选中寄存器。但 IS_WRITE 是个电平：连续两条 WRITE 时它会连着两拍都为 1、中间没有下降沿 ⟹ 只有 1 个上升沿 ⟹ 第二条 WRITE 锁不进去。
修法：WR = IS_WRITE 与 clk（用 与门（AND Gate） 把写信号和时钟相与）。这样每个 WRITE 拍里，clk 上升时 WR 跟着上升一次、下降时落下，每拍一个干净脉冲；连续 WRITE 也因为 clk 每拍都落一次而各自成沿。这是有意为之的「门控时钟脉冲」，仿真完全可靠。

M1

接出 WRITE 写口

在 CPU 子电路放三个输出 引脚（Pin）：类型（Type）=Output，reg_id(4)、value(8)、wr(1)。
• reg_id ← 隧道（Tunnel）REGID；  value ← VAL。
• wr ← 一个 与门（AND Gate）：输入 IS_WRITE 和 clk。

5.2 END：PC 归零、整曲循环

M2

把 PC 的「加载」接到 IS_END

回到 §2 的 PC 计数器，把它的 加载（Load）脚从临时的常量 0 改接 IS_END（数据 D 仍是 16 位常量 0）。这样读到 END(0xF000) 那拍：加载优先 ⟹ PC ← 0，下一拍从曲首重放，无缝循环。
检查一致性：END 不是 WAIT，stall=0、PC 的 计数使能=NSTALL=1，但加载优先于计数，所以装 0 而不是 +1。✔

5.3 frame_commit：一帧写完打一拍，喂 APU 的按帧采样门

M3

用 start 生成 frame_commit

里程碑 2 §4.5 的 frame_clk 当时是「自由跑的慢时钟」，有概率采到半更新的寄存器组合。现在有更好的脉冲源：CPU 每帧把本帧所有 WRITE 写完、正要进入 WAIT 的那一刻 = start（is_wait 且 ¬busy）。这一刻寄存器组已就绪，正适合采样进蜂鸣器（Buzzer）。
再放一个 与门（AND Gate）：frame_commit = START 与 clk（和 WR 同理，做成干净脉冲）。引一个输出 引脚（Pin）frame_commit(1)。联调时把它接到 APU 的 frame_clk。

CPU 子电路对外引脚清点

到这里 CPU 子电路对外应有：输入 clk(1)、reset(1)；输出 reg_id(4)、value(8)、wr(1)、frame_commit(1)。（可选再引出 PC(16)、INSTR(16) 到 探针（Probe）/数码显示，答辩时好看。）存盘。

6接上里程碑 2 的 APU，听 track00

CPU 子电路验完，回到 cpu.circ 的 main 当测试台，把 APU 拉进来联调。

P1

把 apu.circ 当库加载进来

在 cpu.circ 工程里：项目 → 加载库 → Logisim-evolution 库…（Project → Load Library → Logisim-evolution Library…），选 apu.circ。左侧浏览器会多出 apu 库，里面有里程碑 2 的 APU 子电路。（这正是最终 top.circ 组装 CPU+APU 的同一招，提前用上。）

P2

在 main 里摆好 CPU + APU + 时钟

各拖一个 CPU、APU 到 main。放一个 时钟（Clock）（线路 → 时钟（Wiring → Clock））接 CPU.clk；放一个 按钮（Button）（或 1 位 引脚（Pin））接 CPU.reset。
把 CPU.clk 同时也接到 APU 的 WR 之外它需要的同步时钟吗？——不用。APU 的 WR 由 CPU 的 wr 提供（已是 is_write·clk 脉冲）；APU 内部按帧采样用的是 frame_clk，由 CPU 的 frame_commit 提供。CPU 和 APU 共享同一个 时钟（Clock）只是因为 wr/frame_commit 里那个 clk 就是它。

P3

CPU 输出→ APU 输入含义

用 隧道（Tunnel）按名字连最省事。确认全程无红线（接错/悬空）、无灰线（位宽不符）。

P4

开自动节拍，播放

勾 仿真 → 自动传播（Simulate → Auto-Propagate） 和 仿真 → 启用自动节拍（Simulate → Auto-Tick Enabled），仿真 → 自动节拍频率（Simulate → Auto-Tick Frequency） 设 2kHz（不够顺再上 4kHz；务必 >1000Hz，否则被 Windows 定时器精度卡到 ~32 拍/秒）。先按一下 reset 回到曲首再放开。
应当听到 track00（Monitoring）正常播放：开头一串 WRITE 建好音色、reg9=0x0f 开声道后旋律响起，每帧 WAIT 维持音乐帧率，读到 END 自动循环。

判定标准（本里程碑「出音乐」的门槛）
  • 会自己跑：开自动节拍后 PC 持续推进、不卡在第一条 WAIT（= §4 状态机正确，没有死循环）。
  • 节奏稳：旋律不忽快忽慢；调 自动节拍频率 能整体变快/变慢（验证 WAIT 真的在分频）。
  • 音对：能听出是 track00 的曲子（不是噪声/乱跳），声道开关（reg9）生效。
  • 会循环：放到曲尾（END）能无缝从头再来；按 reset 立刻回曲首。
  • 全程无红线 / 无灰线；存盘并备份 cpu_m3.circ。

四条都过 = CPU 自动读谱 → APU 发声 这条主链路打通。这是整个项目的保底验收 + 最大亮点。

卡顿/不出声的排查顺序

① 不出声但 PC 在走：多半是 frame_commit 没接对——没有按帧采样脉冲，蜂鸣器（Buzzer）前那排采样 寄存器（Register） 永不更新。先确认 frame_commit 在每帧有上升沿。② 卡在第一条 WAIT：回 §4 检查 busy 触发器的 D 是否接了 BUSYNEXT、PC 计数使能是否改成了 NSTALL（别还连着常量 1）。③ 第二条 WRITE 丢失/音色不对：检查 wr 是不是 IS_WRITE·clk（不是直接接 IS_WRITE）。④ 整体卡顿/爆音：节拍 >1000Hz 了吗？4 路同响吃力就先只开 1–2 路（里程碑 2 的降级策略），先保 CPU 正确。

  • 译码：两个 Splitter（分线器）拆出 OPCODE/REGID/VAL/IMM，译码器（Decoder，4→16）出 IS_WAIT/IS_WRITE/IS_END。
  • ★ WAIT 停顿状态机：busy 触发器 + 向下 计数器（Counter） + start/en_count/busy_next 逻辑；不死循环、PC 在停顿后正确放行（按 §4.3 表自检）。
  • WRITE：wr = IS_WRITE·clk 写脉冲；END：PC 加载 0 循环。
  • frame_commit = start·clk 引出，联调时接 APU 的 frame_clk。
  • 联调：加载 apu.circ 库，CPU 4 根线接 APU，2–4kHz 自动节拍下 track00 正常播放、节奏稳、会循环。

做完这一步，你已经具备了什么

一颗能自己读谱、按时值发声的单周期五段 CPU，加上里程碑 2 的 APU，真 NES 音乐已经在播了。这就是项目的核心骨架。剩下的都是加分项与老师的硬性要求：把 ALU 接进 WAIT 做可控变速、R0–R3 可视化、控制面板与 16 首切换——见下一里程碑。

8下一里程碑预告

里程碑 4 = ALU 变速播放 + R0–R3 + 控制面板 + 16 首切换（把 CPU 补成完整规格、满足老师硬性要求）：

  • ALU 接进 WAIT 装载路径（老师硬性要求「可控加减」，不可砍）：在 §4 那条「IMM → WAIT 计数器数据脚」的路上插一个 ALU——effective_wait = imm ± tempo_delta，控制面板选 normal/slow/fast。这样真实播放时 ALU 每帧都在参与节奏，不是孤立 demo。顺便把 §4.3 的 −2 精确化也在这里做。
  • LOADI / ADD / SUB / WRITEREG + R0–R3：补齐 ISA 的 ALU 指令与 4 个通用寄存器，配一段手写 demo ROM 现场显示 R2=R0+R1（16 进制数码显示），作为 ALU 可视化证明。
  • 控制面板（角色 A，top.circ）：播放/暂停（门控 CPU 时钟）、停止（reset + 写 reg9=0 静音）、上一首/下一首（4 位曲目计数器 → 16:1 多路复用器（Multiplexer） 选哪个 trackNN.txt ROM）、MM:SS 计时、到时自动下一首。
  • LWAIT（0x2，2 字长等待）：本里程碑 track00 用不到；换某些长静音的曲子才需要，届时给状态机加一个「取下一字当 16 位等待数」的小分支。
