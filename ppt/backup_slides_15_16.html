项目结论（三点）
  • 输入真实：真实 NSF 文件；GME 运行真实播放逻辑，hook 的是寄存器写入，不是音频采样。
  • 执行真实：Logisim 里由 CPU 取指执行、WAIT 控时序、ALU 变速——不是脚本代播。
  • 声音可信：音高来自 NES timer 公式查表（真实 Hz），节奏来自 frame_cycles × WAIT 停顿。
主动声明的边界
  • DMC 采样通道与扩展音源裁剪；Noise 是 Buzzer 低频近似，非完整 LFSR 音色。
  • Buzzer 参数变化会重建音频缓冲——强颤音/包络可能有爆破感，已用 frame_commit + 脚本节流缓解。
  • Windows 上 Logisim 低频自动节拍不准 → 项目固定用 >1000Hz 自动节拍 + WAIT / 电路分频。

16备份页 · Q&A 仅被追问时使用

常见追问 · 一句话回答

为什么不实现完整 6502 / 2A03？
课程目标是 CPU + APU 播放器，不是 NES 模拟器。复杂执行离线交给 GME，Logisim 只实现播放所需的指令、时序与声道。
为什么 Buzzer 频率不按仿真时钟缩放？
Buzzer 按真实墙上时间生成波形；时钟只影响 CPU 速度（节奏）。音高必须给真实 Hz，否则变速会连音高一起变。
frame_cycles=32 凭什么控制节奏？
编码器保证每帧恒占 32 周期（WRITE + 补 WAIT）；CPU 遇 WAIT 真的停 PC。没有 WAIT 停顿，这个数字毫无意义。
$4015 为什么映射到 reg9？
reg_id 仅 4 位（16 槽），$4015 在范围外；借真实 NES 未使用的 $4009 槽位承载声道使能，$4009/$400D 原写入丢弃。
为什么 wr = is_write · clk？
APU 把 WR 当时钟边沿。连续两条 WRITE 时电平不回落，第二条会丢失边沿；与 clk 相与保证每条 WRITE 一个干净脉冲。
ALU 是不是只是演示？
不是。ALU 在 WAIT 装载路径上，真实播放每帧都计算 imm±Δ；slow/fast 现场可听。R0–R3 的 ADD/SUB 是额外证明。
Noise 为什么不像原曲？
NES Noise 是 LFSR 模型，Buzzer 白噪声是按 Hz 平铺随机块；项目做 16 档低频近似保留鼓点层次，不承诺还原音色。
为什么最终 4 首而不是 16 首？
16 个大 ROM 过重且有空槽问题；最终 4 首连续槽位 + 2-bit 曲号 + 4:1 多路复用器（Multiplexer），功能等价且更稳。
